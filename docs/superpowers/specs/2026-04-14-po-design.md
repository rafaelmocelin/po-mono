# po — Design Spec

**Date:** 2026-04-14
**Source plan:** `po-build-plan.md`
**Approach:** Minimal fork (Approach A) — pi core untouched except one compaction hook; all po-specific behavior lives in the extension system.

---

## 1. Architecture Overview

### 1.1 Fork Structure

Fork `badlogic/pi-mono` at latest stable tag. Add it as `upstream` remote for periodic merges.

**Deleted packages:** `pi-web-ui`, `pi-slack`, `pi-vllm`

**Kept:** `pi-coding-agent`, `pi-ai`, `pi-agent-core`, `pi-tui`, package manager, RPC mode, HTML export

**Renamed (pi → po):**
- Binary entrypoint: `pi` → `po`
- Package names: `@mariozechner/pi-coding-agent` → `po-coding-agent` (etc.)
- Config dirs: `~/.pi/agent/` → `~/.po/agent/`, `.pi/` → `.po/`
- Temp file prefixes and any displayed strings referencing "pi"
- `AGENTS.md` convention: unchanged

### 1.2 Core File Modifications (minimal)

Only one file gets a meaningful edit:
- `src/core/agent-session.ts` — add `onBeforeCompact` callback hook so the pipeline can register step-specific preservation lists. Additive only.

These files are **not edited** — po uses their existing extension points instead:
- `src/core/bash-executor.ts` / `tools/bash.ts` — RTK uses custom `BashOperations` injection
- `src/core/system-prompt.ts` — caveman uses existing `appendSystemPrompt` option

### 1.3 Pipeline Module Layout

```
packages/coding-agent/src/pipeline/
├── pipeline.extension.ts      # Extension entry point — registers all slash commands and hooks
├── orchestrator.ts            # Pipeline state machine
├── state.ts                   # Read/write .po/pipeline-state.json
├── step-runner.ts             # Creates per-step agent sessions via pi SDK
├── summary-bridge.ts          # Haiku cross-step summaries
├── cost-tracker.ts            # Per-step, per-model cost accounting
├── status-bar.ts              # Pipeline status bar updater
├── caveman.ts                 # Caveman prompt constant
├── model-tiers.ts             # Tier system, fallback chain, 429 handling
├── rtk-operations.ts          # Custom BashOperations wrapping RTK
└── skills/
    ├── grill-me.md
    ├── write-a-prd.md
    ├── design-an-interface.md
    ├── prd-to-issues.md
    ├── tdd.md
    ├── improve-codebase-architecture.md
    └── references/
        ├── tests.md
        ├── mocking.md
        ├── deep-modules.md
        ├── interface-design.md
        └── refactoring.md
```

`pipeline.extension.ts` is the single pi extension loaded at startup. All other files are modules it imports.

## 2. Phase 1: Foundation

### 2.1 Binary Rename

Full string scan of all source files for `pi` references in config paths, temp file names, displayed strings, package names. `config.ts` updated for `~/.po/` and `.po/` dirs.

### 2.2 RTK Integration — `rtk-operations.ts`

Implements the `BashOperations` interface from `tools/bash.ts`. Its `exec` method prepends `rtk` to every command before delegating to `createLocalBashOperations().exec`.

`--no-rtk` session flag swaps back to bare `createLocalBashOperations()`.

The RTK `BashOperations` is injected at po startup for **all sessions** (pipeline and vanilla), not just pipeline mode. For user-initiated bash (`!` prefix), `pipeline.extension.ts` subscribes to the `user_bash` event and rewrites through RTK before execution. Same `--no-rtk` bypass applies to both.

`po setup` command: checks `which rtk`, `which gh`, `which git`. Prints installation instructions for anything missing. Does not auto-install.

### 2.3 Model Tier System — `model-tiers.ts`

Reads `~/.po/settings.json` on startup. Three tiers:

```json
{
  "tiers": {
    "tier1": { "role": "planning",        "models": ["claude-opus-4-6", "o3", "gemini-2.5-pro"] },
    "tier2": { "role": "implementation",  "models": ["claude-sonnet-4-6", "gpt-4.1", "gemini-2.5-flash"] },
    "tier3": { "role": "mechanical",      "models": ["claude-haiku-4-5", "gpt-4.1-mini", "gemini-2.5-flash-lite"] }
  },
  "escalation": {
    "tier2_max_attempts": 3,
    "tier1_max_attempts": 2
  }
}
```

`resolveModel(tier)` returns first available model in tier chain.
`nextModelInTier(tier, current)` advances on 429. If chain exhausted, exponential backoff on first model.

Extension subscribes to pi's API error events to detect 429s. Status bar shows fallback state: `model: gpt-4.1 (fallback from sonnet-4.6)`.

Per-step `--model` override stored in pipeline state, passed to step runner at session creation. Pi's `Ctrl+P` / `Ctrl+L` model cycling untouched for vanilla po usage.

## 3. Phase 2: Pipeline Infrastructure

### 3.1 `orchestrator.ts`

State machine: `idle → grill-me → write-a-prd → [design-an-interface] → prd-to-issues → tdd → done`

`design-an-interface` skipped by default. Triggered by `--design` flag or grill-me recommendation + user confirmation.

Exposed methods: `advance()`, `jumpTo(step)`, `jumpToIssue(n)`. On `--resume`, reads `.po/pipeline-state.json`, identifies current step and issue, checks out the issue branch, runs the project test suite to determine which tests currently pass, then continues from the last green commit.

### 3.2 `state.ts`

Reads/writes `.po/pipeline-state.json`. Atomic writes (write to `.tmp`, rename) to prevent corruption on crash.

Schema:
```json
{
  "pipeline_id": "uuid",
  "started_at": "ISO8601",
  "current_step": "tdd",
  "current_issue": "003",
  "steps_completed": [
    {
      "step": "grill-me",
      "completed_at": "...",
      "artifact": "docs/agent/grill-me-decisions.md",
      "cost": { "input_tokens": 0, "output_tokens": 0, "model": "claude-opus-4-6", "usd": 0.0 }
    }
  ],
  "issues": [
    { "number": 1, "github_issue": 42, "status": "done", "branch": "agent/issue-001-auth-setup" }
  ]
}
```

### 3.3 `step-runner.ts`

Creates a fresh agent session per step using pi's `createAgentSession` SDK. Assembles initial context:
1. Step skill prompt (from `skills/*.md`)
2. Summaries from prior steps (`docs/agent/summaries/step-N-summary.md`)
3. Artifact file references (read on demand by agent)
4. Project `AGENTS.md`

Injects caveman via `appendSystemPrompt` when `caveman: true` in skill frontmatter. Injects RTK `BashOperations`. Injects per-step model from tier resolution (or explicit `--model` override).

Returns when step produces its output artifact or user types `/next`.

### 3.4 `summary-bridge.ts`

After step completion, creates a Haiku (tier 3) session with caveman ON. Feeds step session transcript. Outputs structured summary to `docs/agent/summaries/step-N-summary.md`:
- Key decisions made
- Constraints identified
- Deferred items
- References to artifact files

This summary — not the raw transcript — seeds the next step's context.

### 3.5 `caveman.ts`

Single exported constant:

```
"Terse like caveman. Technical substance exact. Only fluff die. Drop: articles, filler (just/really/basically), pleasantries, hedging. Fragments OK. Short synonyms. Code unchanged."
```

Step runner appends this to `appendSystemPrompt` when caveman is ON.

### 3.6 `cost-tracker.ts`

Subscribes to pi's `message_end` events. Captures input/output token counts per session. Maps to USD using static rate table. Writes per-step totals to `pipeline-state.json` when step completes.

`/costs` reads the state file and renders the table:
```
Pipeline: prd-001 | Total: $2.34

Step                  Model           Input    Output   Escalations  Cost
─────────────────────────────────────────────────────────────────────────
grill-me              opus-4.6        12.0k    4.5k     —            $0.45
...
RTK savings this session: ~48,000 tokens saved (est. $0.80)
```

### 3.7 Step-Aware Compaction

Each skill's `.md` frontmatter declares a `preserve_on_compaction` list. Step runner reads this on load and registers an `onBeforeCompact` callback (the hook added to `agent-session.ts`) that pins those items. Haiku handles summarization of non-preserved content.

Per-step preservation defaults:

| Step | Preserved |
|---|---|
| grill-me | Decision tracker (open/resolved branches) |
| write-a-prd | PRD draft, module list, test strategy |
| design-an-interface | Interface candidates, comparison matrix |
| prd-to-issues | Issue list with blocking relationships |
| tdd | Test plan, current slice, remaining slices, current test file |
| improve-architecture | Findings list, improvement candidates |
| summary-bridge (Haiku) | Nothing — brevity is the point |

### 3.8 Slash Commands

Registered by `pipeline.extension.ts` via `registerCommand()`:

| Command | Handler |
|---|---|
| `/pipeline [--from/--step/--issue/--design/--resume]` | orchestrator |
| `/grill-me` | step-runner standalone |
| `/tdd` | step-runner standalone |
| `/improve-architecture` | step-runner standalone |
| `/board` | reads local issue cache, renders board |
| `/costs` | reads pipeline-state.json, renders cost table |
| `/next` | signals current step to advance |
| `/rtk-stats` | calls `rtk gain` via bash, displays output |

### 3.9 Status Bar

During pipeline execution, `pipeline.extension.ts` calls `setStatus("pipeline", ...)` after each state transition:

```
[PIPELINE: prd-001] [STEP: tdd #003 api-routes] [MODEL: sonnet-4.6] [3/8 issues | blocked: #6,#7] [$2.34]
```

Cleared when no pipeline is running. Vanilla po footer unchanged outside pipeline mode.

## 4. Phase 3: Skills

### 4.1 Skill Frontmatter Schema

All skill files use YAML frontmatter:

```yaml
---
name: grill-me
tier: 1
caveman: false
input_artifacts: []
output_artifact: docs/agent/grill-me-decisions.md
preserve_on_compaction:
  - Decision tracker (open/resolved branches)
---
```

### 4.2 Skill Sources and Additions

**`grill-me.md`** — Ported from `mattpocock/skills/grill-me/SKILL.md`. Added: dynamic decision tracker format (`[RESOLVED]/[OPEN]/[DEFERRED]` tree displayed during session), codebase exploration before asking, termination condition (all RESOLVED), `/next` to confirm.

**`write-a-prd.md`** — Ported from `mattpocock/skills/write-a-prd/SKILL.md`. Reads `grill-me-decisions.md`. Added: "no file paths or code snippets in Implementation Decisions" constraint, explicit deep-module sketching step, artifact saved to `docs/agent/prd-NNN.md`.

**`design-an-interface.md`** — Ported from `mattpocock/skills/design-an-interface/SKILL.md`. Reads `prd-NNN.md`. Output to `docs/agent/interface-designs-NNN.md`. User selection noted in artifact.

**`prd-to-issues.md`** — Ported from `mattpocock/skills/prd-to-issues/SKILL.md`. Reads local `prd-NNN.md` (not a GitHub issue). Added: `gh issue create` write path, local Markdown cache at `docs/agent/issues/NNN-slug.md` with YAML frontmatter. HITL/AFK classification preserved.

**`tdd.md`** — Ported from `mattpocock/skills/tdd/SKILL.md` (comprehensive vertical-slice TDD). Added: reads issue file + PRD test strategy as input; branch creation (`agent/issue-NNN-slug`); one commit per green test with `[issue-NNN]` prefix; pause between issues; `gh issue close` + local cache update on completion. References supplementary files in `references/` by relative path.

Escalation mechanism: the tdd skill itself does not switch models. After 3 failed green attempts on a slice, the skill emits a structured signal in its output (e.g., `ESCALATE: slice-description`). The step runner detects this signal, terminates the current tier 2 session, and spawns a new sub-session using the tier 1 model for that sub-task only. After 2 more failures in the escalated session, the step runner pauses and prompts the user for guidance.

**`improve-codebase-architecture.md`** — Ported from `mattpocock/skills/improve-codebase-architecture/SKILL.md`. On-demand via `/improve-architecture` only. No pipeline wiring needed.

**Supplementary files** — `tests.md`, `mocking.md`, `deep-modules.md`, `interface-design.md`, `refactoring.md` fetched verbatim from `mattpocock/skills` and placed in `src/pipeline/skills/references/`.

## 5. Phase 4: GitHub Integration

**Authentication:** `gh auth status` checked by `po setup`. All mutations go through `gh` CLI (no direct API calls).

**Write path:** `gh issue create`, `gh issue close`, `gh issue edit`, `gh issue comment`

**Read path:** Local cache `docs/agent/issues/` is primary. Fast, offline, no API calls during TDD.

**Local cache format** (`docs/agent/issues/NNN-slug.md`):
```yaml
---
number: 1
github_issue: 42
title: Auth setup with JWT
status: backlog
blocked_by: []
priority: 1
branch: null
---
```

**Sync on pipeline start:** `gh issue list --json number,state,labels,title` fetches current state. Local cache updated to match. GitHub is source of truth — local changes overwritten.

**`/board`** reads local cache only (no network). Renders four-column terminal view: BACKLOG / IN PROGRESS / DONE / BLOCKED.

**Git branching:** One branch per issue: `agent/issue-NNN-slug`, created from current branch. One commit per green test. Po does not auto-merge — pauses and surfaces branch name for user.

## 6. Phase 5: Polish & Testing

Steps 29–35 from the build plan are tuning passes, not pre-specified code changes. They run after the first full pipeline execution on a real project. Findings drive adjustments to: escalation thresholds, compaction preservation lists, caveman prompt, cross-step summary quality.

Concrete output: `AGENTS.md` for po itself (written at end of Phase 5).

## 7. Upstream Merge Strategy

`badlogic/pi-mono` added as `upstream` remote. Periodic `git fetch upstream && git merge upstream/main`. Expected conflicts limited to:
- `package.json` (name/version)
- Binary entrypoint
- `agent-session.ts` (the `onBeforeCompact` hook)

All other pipeline code lives in `src/pipeline/` — no upstream conflicts there.
